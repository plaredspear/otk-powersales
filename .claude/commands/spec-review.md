# 스펙 리뷰 모드

스펙 문서의 AI(Sonnet 4.6) 구현 준비도를 자동 검증합니다.
Part 파일 분할 적합성도 함께 평가합니다.

## 입력 파라미터 분석

사용자가 전달한 인자를 파싱합니다: $ARGUMENTS

인자는 다음 형태 중 하나입니다. 형태를 판별하여 적절히 처리합니다:

| 형태 | 예시 | 처리 방식 |
|------|------|----------|
| **인자 없음** | `/spec-review` | 사용자에게 리뷰 대상을 질문 |
| **폴더명** | `1-redis-config` | `docs/specs/backlog/<폴더명>/`에서 spec.md + P*.md 읽어 리뷰 |
| **스펙 번호** | `1` | `docs/specs/backlog/`에서 `<번호>-*` 폴더를 탐색하여 리뷰 |

### 판별 규칙

1. 숫자만이면 → **스펙 번호** → `docs/specs/backlog/`에서 `<번호>-*` 폴더 탐색
2. `숫자-텍스트` 패턴이면 → **폴더명** → `docs/specs/backlog/<폴더명>/` 직접 접근
3. 인자가 없으면 → 사용자에게 질문:
   - "리뷰할 스펙을 지정해주세요. (폴더명 또는 스펙 번호)"

### 스펙 로드 절차

1. `docs/specs/backlog/` 하위에서 대상 폴더 탐색 (없으면 `docs/specs/ready/`도 확인)
2. `spec.md` 읽기 (부모 스펙, 전체 컨텍스트)
3. `P*.md` 파일 모두 읽기 (Part별 스펙)
4. spec.md에서 플랫폼 정보 추출

## 리뷰 실행

### 플랫폼/작업유형 판별

스펙 내용에서 플랫폼과 작업 유형을 자동 판별합니다:

- **플랫폼**: 파일명 접미사, 개요 테이블의 플랫폼 필드, 또는 Issue 라벨
- **작업 유형**: 파일명 접두사 (F=Feature, R=Refactor, S=System, B=Bug) 또는 Issue 제목

### spec-reviewer 에이전트 호출

Task 도구로 `spec-reviewer` 에이전트를 호출합니다:

```
에이전트에 전달할 프롬프트:

다음 스펙의 AI 구현 준비도를 리뷰해주세요.

- 플랫폼: <판별된 플랫폼>
- 작업 유형: <판별된 작업 유형>
- 스펙 내용:

<로드된 스펙 내용>
```

에이전트가 `.claude/guides/spec-review-criteria.md`를 읽고 리뷰를 수행합니다.

## 결과 출력 및 후속 안내

에이전트로부터 리뷰 리포트를 수신하면 사용자에게 표시하고,
종합 판정에 따라 후속 단계를 안내합니다:

### READY 판정 시

1. 리뷰 리포트를 터미널에 표시합니다
2. **스펙 파일의 승인 이력에 AI 리뷰 결과를 즉시 기록합니다**:
   - 스펙 파일(spec-B.md 또는 spec.md)의 승인 이력 테이블에서 AI 리뷰 행을 업데이트
   - 상태: `READY`, 승인자: `Claude`, 날짜: 당일, 비고: `FAIL N / WARN N` (해당 수치 기재)
   - 예: `| AI 리뷰 | READY | Claude | 2026-02-23 | FAIL 0 / WARN 2 (Refactor 특성) |`
3. **사용자에게 스펙 리뷰를 요청합니다**:

```
리뷰 결과: READY - AI 리뷰 통과

스펙 폴더를 에디터에서 최종 확인해주세요:
docs/specs/<번호>-<기능명>/

확인 후 터미널에서 승인 또는 수정 의견을 알려주세요.
```

4. 사용자 응답 대기:
   - **승인** ("승인", "OK", "진행" 등) → 사람 리뷰 승인 이력 업데이트 → **스펙 폴더를 `docs/specs/backlog/`에서 `docs/specs/ready/`로 이동** → 구현 시작 안내
   - **수정 요청** (구체적 피드백) → 스펙 파일 수정 후 재리뷰 안내

### NEEDS_WORK 판정 시

1. 리뷰 리포트를 표시합니다
2. WARN 항목을 **자동 수정 가능 / 사용자 확인 필요**로 분류합니다
3. **자동 수정 가능한 항목은 즉시 스펙 파일에 반영합니다**:
   - 참조 패턴 누락 → 코드베이스에서 동일 패턴 파일을 검색하여 참조 패턴 섹션 추가
   - 에러 코드 테이블 누락 → 기존 SAP API 패턴을 참조하여 에러 코드 테이블 생성
   - 유효성 검증 규칙 미흡 → 요청 필드 테이블에 검증 규칙 컬럼 추가
   - Phase2 활성화 대상 미식별 → 코드베이스에서 Phase2 파일 검색 후 결과 기재
   - 레거시 스키마 제약 검토 부족 → Heroku DB 스키마를 조회하여 기존 컬럼 존재 여부 확인 후 기재
4. **사용자 확인이 필요한 항목은 선택지와 함께 질문합니다**:
   - 비즈니스 결정이 필요한 항목 (예: "구현 시 결정한다" 같은 미확정 사항)
   - 여러 대안 중 선택이 필요한 항목
   - 예: "SystemCodeMaster 처리 방식을 확정해주세요: (A) 기존 엔티티 사용, (B) enum 매핑, (C) application.yml 설정"
5. 자동 수정 완료 + 사용자 응답 반영 후 **자동으로 재리뷰를 실행합니다**

#### 자동 수정 분류 기준

| 분류 | 조건 | 예시 |
|------|------|------|
| **자동 수정** | 코드베이스 검색으로 확인 가능한 사실 기반 항목 | 참조 패턴, Phase2 존재 여부, 레거시 컬럼 매핑, 에러 코드 패턴, 유효성 규칙 |
| **사용자 확인** | 비즈니스 결정/설계 선택이 필요한 항목 | 코드 변환 실패 정책, 구현 방식 선택, 최대 건수 제한값 |

### NOT_READY 판정 시

1. 리뷰 리포트를 표시합니다
2. FAIL 항목 중 **자동 수정 가능한 항목은 즉시 반영합니다** (위 NEEDS_WORK와 동일 기준)
3. 사용자 확인이 필요한 FAIL 항목은 선택지와 함께 질문합니다
4. 자동 수정 완료 + 사용자 응답 반영 후 **자동으로 재리뷰를 실행합니다**
