# 스펙 작성 가이드

이 문서는 스펙(Spec) 작성 시 Claude가 따라야 할 규칙을 정의합니다.
`spec-review-criteria.md`는 사후 검증(채점) 기준이고, 이 문서는 **작성 시점의 지침**입니다.

---

## 핵심 원칙

1. **스펙만으로 구현 가능** — 구현자가 스펙 + 컨벤션 문서(`backend-conventions.md`, `mobile-conventions.md`)만으로 코드를 작성할 수 있어야 한다
2. **코드는 쓰지 않는다** — 자연어와 데이터 테이블로만 서술한다
3. **크기를 통제한다** — Part당 800줄 이내를 목표로 한다

---

## 절대 금지사항

### 코드 블록 금지

다음 언어의 코드 블록을 포함하지 않는다:
- Java, Kotlin, Dart, TypeScript, SQL(DDL/DML), HCL(Terraform)

### 허용되는 코드성 표현

| 허용 항목 | 용도 | 예시 |
|----------|------|------|
| JSON 예시 | API 요청/응답 계약 | `{ "employeeId": "12345", "name": "홍길동" }` |
| 설정 키-값 테이블 | 환경 변수, 설정값 정의 | `REDIS_TTL_SECONDS: 300` |
| ASCII 와이어프레임 | 화면 레이아웃 | 텍스트 박스 기반 UI 구조 |
| TypeScript 인터페이스 | 데이터 계약 (W 플랫폼) | `interface User { id: string; name: string; }` |

### 모호한 표현 금지

다음 표현을 사용하지 않는다. 구체적 수치/조건으로 대체한다:
- "적절히", "필요에 따라", "등", "기타", "관련 처리"
- "정상 동작해야 함" → 구체적 기대값으로 대체
- "에러 처리" → 어떤 에러에 어떤 응답을 하는지 명시

### 컨벤션 반복 금지

`backend-conventions.md`, `mobile-conventions.md`에 이미 정의된 내용을 스펙에 재기술하지 않는다:
- ApiResponse 구조, Exception 패턴, 공통 위젯 스타일 등
- 필요하면 "컨벤션 문서의 X 패턴을 따른다"로 참조만 한다

## 스펙 파일 구조

### 파일명 규칙

| 경우 | 파일 구성 |
|------|----------|
| 단일 플랫폼 | `spec-{P}.md` (예: `spec-B.md`, `spec-M.md`) |
| 복수 플랫폼 | `spec.md` (개요 + 공유 모델) + `P{N}-{P}.md` (플랫폼별 상세) |

### Part 분할 기준

- 800줄을 초과하면 분할한다
- 플랫폼이 다르면 반드시 분할한다
- 각 Part는 독립적으로 구현 가능해야 한다 (자기 완결성)

---

## 필수 섹션 (플랫폼 공통)

모든 스펙 파일에 아래 섹션을 포함한다:

### 1. 개요 테이블

```markdown
| 항목 | 내용 |
|------|------|
| 스펙 번호 | #NNN |
| 기능명 | 기능 이름 |
| 플랫폼 | B / M / W / I |
| 상태 | DRAFT |
| Part 수 | N |
```

### 2. 승인 이력 테이블

```markdown
| 단계 | 상태 | 리뷰어 | 일자 | 비고 |
|------|------|--------|------|------|
| AI 리뷰 | PENDING | - | - | - |
| 사람 리뷰 | PENDING | - | - | - |
```

### 3. 배경 / 현재 상태

- 이 기능이 왜 필요한지 (동기)
- 현재 시스템의 상태 (있다면)

### 4. 완료 조건

검증 가능한 체크리스트. 모호한 항목 금지:
- (O) "GET /api/v1/users 호출 시 200과 사용자 목록 반환"
- (X) "사용자 조회가 정상 동작"

### 5. 데이터 모델

4열 테이블 필수:

```markdown
| 필드명 | 타입 | 필수 | 설명 |
|--------|------|:----:|------|
| employeeId | String | Y | 사번 (6자리) |
| name | String | Y | 사원명 |
```

### 6. 영향 범위 / 파일 목록

신규 생성과 기존 수정을 구분한다. 패키지/디렉토리 수준으로 기술한다:
- **신규**: `com.otk.salesforce.domain.order` 패키지
- **수정**: `SecurityConfig` — 새 엔드포인트 권한 추가

### 7. 테스트 시나리오

테이블 형식. Happy Path와 Error Path를 분리한다:

```markdown
| 시나리오 | 조건 | 예상 결과 |
|---------|------|----------|
| 정상 조회 | 유효한 사번으로 GET 요청 | 200, 사용자 정보 JSON 반환 |
| 미존재 사번 | 존재하지 않는 사번 | 404, NOT_FOUND 에러 코드 |
```

### 8. 변경 이력

```markdown
| 버전 | 날짜 | 변경 내용 |
|------|------|----------|
| v1.0 | 2026-03-03 | 초안 작성 |
```

## 플랫폼별 필수 섹션

### B (Backend)

공통 필수 섹션 + 아래 항목:

1. **엔드포인트 정의** — HTTP 메서드 + 경로 + 요청/응답 JSON 예시 + 에러 응답
2. **인증/권한** — 각 API의 인증 필요 여부, 권한 레벨
3. **비즈니스 로직** — 번호 매긴 자연어 목록 (코드 금지)
4. **DTO 필드 정의** — 요청/응답 DTO를 데이터 모델 테이블로 정의
5. **기존 코드 영향 분석** — Phase2 활성화 대상, 공유 컴포넌트 변경 여부, 레거시 엔티티 제약

### M (Mobile)

공통 필수 섹션 + 아래 항목:

1. **화면 설계** — ASCII 텍스트 와이어프레임 (실제 데이터 예시 포함)
2. **화면-API 매핑 테이블** — 화면 동작과 호출 API 매핑
3. **비즈니스 로직** — 번호 매긴 자연어 목록 (코드 금지)
4. **스코프 외 명시** — 이 스펙에서 다루지 않는 항목 명시

### W (Web)

공통 필수 섹션 + 아래 항목:

1. **화면 설계** — ASCII 와이어프레임 + UI 컴포넌트 목록
2. **사용자 흐름** — 화면 간 네비게이션/흐름 정의
3. **컴포넌트 설계** — 페이지/훅/API 함수 목록 (패키지 수준)
4. **TypeScript 인터페이스** — 데이터 계약용 인터페이스 정의

### I (Infra)

공통 필수 섹션 + 아래 항목:

1. **변경 대상 모듈** — 모듈명 + 리소스별 설정값
2. **환경별 설정 매트릭스** — dev/staging/prod 설정 차이
3. **비용 영향** — 추가 비용 예상 또는 "비용 변동 없음" 근거
4. **롤백 방안** — 롤백 가능 여부 + 데이터 손실 위험

## 서술 규칙

### 비즈니스 로직

번호 매긴 목록으로 작성한다. 각 항목은 **주어 + 동사 + 목적어** 구조:

```
1. 서비스가 employeeId로 사원 정보를 조회한다
2. 사원이 존재하지 않으면 NotFoundException을 던진다
3. 조회된 사원의 소속 조직 코드로 팀 목록을 조회한다
```

### 데이터 모델

반드시 4열 테이블(필드명 | 타입 | 필수 | 설명)을 사용한다. 타입 없는 필드 목록 금지.

### 테스트 시나리오

- Happy Path와 Error Path를 분리한다
- 기대값은 정량적으로 작성한다 (HTTP 상태 코드, 에러 코드, 반환 건수 등)
- "정상 동작" 같은 추상적 기대값 금지

### 와이어프레임

ASCII 박스로 레이아웃을 표현하고 실제 데이터 예시를 포함한다:

```
┌─────────────────────────┐
│ 홈 화면                   │
├─────────────────────────┤
│ 사번: 123456              │
│ 이름: 홍길동               │
├─────────────────────────┤
│ [실적 조회]  [목표 관리]    │
└─────────────────────────┘
```

## 자체 검증 체크리스트

스펙 작성 완료 후, `/spec-review` 실행 전에 아래 10개 항목을 확인한다:

| # | 항목 | 확인 |
|---|------|------|
| 1 | Java/Kotlin/Dart/TypeScript/SQL/HCL 코드 블록이 없는가? | |
| 2 | "적절히", "필요에 따라", "등" 같은 모호한 표현이 없는가? | |
| 3 | 모든 데이터 모델이 4열 테이블(필드명/타입/필수/설명)인가? | |
| 4 | 테스트 시나리오에 정량적 기대값이 있는가? ("정상 동작" 금지) | |
| 5 | Part당 800줄 이내인가? | |
| 6 | 개요 테이블과 승인 이력 테이블이 있는가? | |
| 7 | 완료 조건이 검증 가능한 체크리스트인가? | |
| 8 | 컨벤션 문서 내용을 재기술하지 않았는가? | |
| 9 | 신규 생성 / 기존 수정 파일이 구분되어 있는가? | |
| 10 | Happy Path + Error Path 테스트가 모두 있는가? | |
